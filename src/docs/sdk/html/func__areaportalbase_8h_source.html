32  // This is called each frame for each client to all portals to close
33  // when the viewer is far enough away, or on the backside.
34  //
35  // The default implementation closes the portal if the viewer (plus some padding)
36  // is on the backside of the portal. Return false if you close the portal.
37  //
38  // Returns whether or not the (server part of) the engine was told to shut the
39  // portal off for purposes of flowing through portals to determine which areas to
40  // send to the client.
41  //
42  //
43  // bIsOpenOnClient is usually the same as the return value but NOT always. Here's why (explained
44  // here in depth because this case was discovered in a lengthy debugging session):
45  //
46  // - Each CFuncAreaPortalBase represents two dareaportal_t's (matched by CFuncAreaPortalBase::m_portalNumber
47  // and dareaportal_t::m_PortalKey). Each dareaportal_t leads into one of the connected areas
48  // and the dareaportal_t's have opposite-facing planes.
49  //
50  // - The engine's SetAreaPortalState function takes a portal key and closes BOTH dareaportal_t's associated with it.
51  //
52  // - UpdateVisibility may decide a portal leading out of the _area you're sitting in_ can be closed
53  // for purposes of flowing through areas because you're on the backside of the dareaportal_t that
54  // you would flow out of.
55  //
56  // - At the same time, you might be able to look through the other dareaportal_t attached to
57  // that portal key (right back into the area you're standing in).
58  //
59  // - An illustration:
60  //
61  // --------------------
62  // | |
63  // | |--------|aaaaaa|
64  // | | |bbbbbb| <---- aaaa and bbbb area both for PortalKey1
65  // |**| | |
66  // | | | area |
67  // |**| | 2 |
68  // | | |------|
69  // | | | |
70  // | ---------- area |
71  // | 1 |
72  // |------------------|
73  //
74  // "aaaa" and "bbbb" each represent a different dareaportal_t, (aaa leads into area 2 and
75  // bbb leads into area 1). They both represent the same portal key though (call it PortalKey1).
76  //
77  // When standing in area 1 (where the "area 1" text is), the engine will check aaaa and it'll notice
78  // that you're on the wrong side of aaaa to be looking through it, so it'll say that PortalKey1 is closed
79  // for purposes of flowing through areas on the server (it turns out you can get to area 2 through
80  // the portal right above the "area 1" text).
81  //
82  // If you told the client that PortalKey1 was closed, then you'd get a pop when moving from area 1
83  // to area 2 because the client would think you couldn't see through PortalKey1 into area 1 UNTIL
84  // the server transmitted the new updated PortalKey bits, which can be lagged.
85  //
86  // That's why we have bIsOpenOnClient which doesn't include this backfacing test.
87  //
88  // .. and they all lived happily ever after.
89  // The End
90  //
91  //
92  //
93  // Note: when you're standing in the space between the **'s, then the server would stop transmitting
94  // the contents of area 2 because there would be no portal you were on the correct side of to
95  // see into area 2.
96  virtual bool UpdateVisibility( const Vector &vOrigin, float fovDistanceAdjustFactor, bool &bIsOpenOnClient );
