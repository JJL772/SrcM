18 // I’m not sure if player spawn cache is the most descriptive name, but essentially
19 // there is a singleton instance of CPlayerSpawnCache for the local player which has
20 // a set of counters/pointers/etc. that get cleared every time a map loads. This
21 // can be useful for situations where a player’s net connection chokes and they get
22 // a full update, which recreates their C_TF_Player entirely or otherwise invalidates
23 // a bunch of data in the local player. I believe it’s already known that there is
24 // a class of bugs stemming from this behavior.
25 //
26 // Right now the cache is used as a way to display a message to the player if they
27 // connect to a server that’s recording replays. As soon as they choose their player
28 // class, a counter is checked, and if it’s zero the message is displayed. The counter
29 // is then incremented. This is a sort of odd use for it actually. A better example
30 // would be what I’m going to do next, which is that if the player’s net connection
31 // chokes (or if you host_timescale at too great a speed and cause a full update on the
32 // client), the replay system will think that you’ve already saved a replay for that life.
33 // So this example will be a perfect time to use the player spawn cache because you can
34 // maintain some level of persistence in the face of your entire local player getting
35 // nuked.
36 //
37 // Just add any data members you'd like to access to the CPlayerSpawnCache::Data_t
38 // struct and it will be cleared automatically (via a memset) whenever a new map is
39 // loaded.
40 //
41 // It's possible that PreReset()/PostReset() or the like will be necessary for this
42 // class to reach its full potential.
43 //
