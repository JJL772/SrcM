21 //-----------------------------------------------------------------------------
22 // Usage model for the UnitTest library
23 //
24 // The general methodology here is that clients are expected to create unit
25 // test DLLs that statically link to the unit test DLL and which implement
26 // tests of various libraries. The unit test application will dynamically
27 // load all DLLs in a directory, which causes them to install their test cases
28 // into the unit test system. The application then runs through all tests and
29 // executes them all, displaying the results.
30 //
31 // *** NOTE: The test suites are compiled in both debug and release builds,
32 // even though it's expected to only be useful in debug builds. This is because
33 // I couldn't come up with a good way of disabling the code in release builds.
34 // (The only options I could come up with would still compile in the functions,
35 // just not install them into the unit test library, or would make it so that
36 // you couldn't step through the unit test code).
37 //
38 // Even though this is the case, there's no reason not to add test cases
39 // directly into your shipping DLLs, as long as you surround the code with
40 // #ifdef _DEBUG. To error check a project to make sure it's not compiling
41 // in unit tests in Release build, just don't link in unitlib.lib in Release.
42 // You can of course also put your test suites into separate DLLs.
43 //
44 // All tests inherit from the ITestCase interface. There are two major kinds
45 // of tests; the first is a single test case meant to run a piece of
46 // code and check its results match expected values using the Assert macros.
47 // The second kind is a test suite which is simply a list of other tests.
48 //
49 // The following classes and macros are used to easily create unit test cases
50 // and suites:
51 //
52 // Use DEFINE_TESTSUITE to define a particular test suite, and DEFINE_TESTCASE
53 // to add as many test cases as you like to that test suite, as follows:
54 //
55 // DEFINE_TESTSUITE( VectorTestSuite )
56 //
57 // DEFINE_TESTCASE( VectorAdditionTest, VectorTestSuite )
58 // {
59 // .. test code here ..
60 // }
61 //
62 // Note that the definition of the test suite can occur in a different file
63 // as the test case. A link error will occur if the test suite to which a
64 // test case is added has not been defined.
65 //
66 // To create a test case that is not part of a suite, use...
67 //
68 // DEFINE_TESTCASE_NOSUITE( VectorAdditionTest )
69 // {
70 // .. test code here ..
71 // }
72 //
73 // You can also create a suite which is a child of another suite using
74 //
75 // DEFINE_SUBSUITE( VectorTestSuite, MathTestSuite )
76 //
